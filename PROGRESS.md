## What was built in session 6 (2026-07-21 — marketing/analytics tracking + order webhook, v2.0.4 → v2.0.5)

Two owner requests, both shipped in one "marketing" commit, released as v2.0.5. Full contract: checkout runs entirely on Shopify's own hosted/custom-domain checkout, **outside this Next.js app** — no page here ever sees a confirmed order client-side. That constraint drives every design choice below.

**Part 1 — client-side ad pixels** (Google Ads + Meta Pixel):
- New optional `config/marketing.ts` (`googleAdsConversionId`, `googleAdsConversionLabel`, `metaPixelId`, all empty-string-means-unset like `config/seo.ts`) + `MarketingConfig` type in `lib/types/config.ts` + `resolveMarketingConfig()` in `lib/config.ts` — same backward-compatible pattern as every other `resolve*Config`.
- `app/layout.tsx` conditionally injects (via `next/script`, `strategy="afterInteractive"` — **first use of `next/script` in this repo**): gtag.js + inline `gtag('config', …)` when `googleAdsConversionId` is set; the Meta Pixel base snippet (`fbq('init', …)` + `fbq('track','PageView')`) when `metaPixelId` is set, plus a new `components/analytics/pixel-page-view.tsx` client component that fires `PageView` again on every client-side route change (App Router navigations don't reload the document, so the base snippet's PageView only covers the hard load — this component skips its own first-mount fire to avoid double-counting that same load).
- New `lib/analytics.ts` — `trackAddToCart`/`trackViewContent`/`trackPageView`/`trackCheckoutConversion`, pure functions that check `window.fbq`/`window.gtag` exist before calling (so call sites never need to check `metaPixelId` is configured themselves — if the script wasn't injected, `window.fbq` doesn't exist and the call is a safe no-op).
- **Because checkout is off-site, there is no true Purchase event anywhere in this codebase.** Two proxy signals instead: Meta Pixel gets `AddToCart` (quick-add button + PDP add/buy-now, both variant-price-aware, fired only after `result.ok`) and `ViewContent` (once per PDP mount) — deliberately no client-side `Purchase`. Google Ads' conversion fires on the "Go to Checkout" `onClick` in all three checkout-link locations (`order-summary.tsx`, `mini-cart.tsx`, `cart-page-content.tsx`'s mobile sticky bar) as a **checkout-intent proxy**, fire-and-continue (no `preventDefault`, never blocks the navigation to Shopify).

**Part 2 — server-side order webhook** (the only place a *confirmed* purchase actually exists in this system):
- `app/api/webhooks/shopify/orders/route.ts` — first API route in the repo (the old `/api/predictive-search` route was removed back in the search-removal pass; see AGENTS.md). `runtime = "nodejs"` (needs `node:crypto`). Reads the raw body via `request.text()` **before** any JSON parsing (HMAC is computed over raw bytes — parsing-then-restringifying would silently break verification), checks `X-Shopify-Hmac-Sha256` against a fresh HMAC-SHA256 keyed with `SHOPIFY_WEBHOOK_SECRET`, `crypto.timingSafeEqual` (constant-time; explicit length check first since it throws rather than returning false on length mismatch), 401 on any failure or missing secret. On success, if `META_CAPI_ACCESS_TOKEN` **and** `resolveMarketingConfig().metaPixelId` are both set, POSTs a real `Purchase` event to Meta's Conversions API (email SHA-256-hashed, never plaintext — Meta requires hashed PII; `event_id` derived from the Shopify order id so Meta's own dedup collapses Shopify's routine webhook retries into one counted Purchase). CAPI failures are logged, never fail the webhook response — Shopify retries aggressively on non-2xx and an optional integration shouldn't cause that. **No server-side Google Ads call** — deliberately out of scope, would need Google Ads API OAuth this app doesn't do.
- `lib/webhooks/verify-shopify-hmac.ts` — the HMAC check pulled out as a pure `verifyShopifyWebhook(rawBody, signature, secret): boolean`, dependency-free, so it's unit-testable in isolation. **No test file was added — this repo has no Vitest/Jest setup at all** (checked; nothing to hook into). Verified by hand instead: wrote a throwaway smoke script in the scratchpad dir exercising valid-sig/wrong-secret/tampered-body/empty-sig/empty-secret/garbage-sig — all six behaved correctly. If a test runner ever gets added to this repo, this file is the natural first unit test.
- `app/(store)/thank-you/page.tsx` — generic order-confirmation page, same static-page pattern as `/contact` (`ProsePage` layout, `metadata` export, all copy through `resolveLegalConfig()`). Reads optional `order_number`/`email` query params and personalizes when present, reads perfectly reasonably without them. **Important: this route has no guaranteed traffic.** Standard (non-Plus) Shopify checkout has no supported way to redirect back to a custom domain after payment — this page exists for Shopify Plus merchants who configure that redirect themselves, or for anything else that might link to it (confirmation emails). `robots: { index: false, follow: false }` since it's a per-customer URL, and it's deliberately not in `app/sitemap.ts`.
- New env vars (both optional, both Vercel-only like `SHOPIFY_ADMIN_TOKEN` — **never** committed to a config file): `SHOPIFY_WEBHOOK_SECRET` (the custom app's API secret key — used to verify the HMAC, *not* the storefront/admin API token) and `META_CAPI_ACCESS_TOKEN`. Added to the `zod` schema in `lib/env.ts` as `.optional()`.

**No `config/` protected-path backfill concern this time** (unlike a hypothetical new required field) — `config/marketing.ts` is a genuinely new file. Stores created *before* v2.0.5 won't have it, and since `config/` is a protected path StoreForge's update engine never writes into on an update, **any pre-v2.0.5 store that updates to this tag (or later) will fail its next build** at `import { marketingConfig } from "@/config/marketing"` in `lib/config.ts`, unless `store-bulk`'s update flow learns to backfill missing (not overwrite existing) files inside protected paths first. Flagged to the owner during the build; **not confirmed done as of this session** — check before recommending anyone update an existing store past v2.0.4. Same backfill mechanism would be needed for any future new config file, so it's worth making generic in `store-bulk` rather than a one-off fix.

Also flagged, owner-side, not build work:
- `store-bulk`'s `render-config.ts` needs to learn to write `config/marketing.ts` (`googleAdsConversionId`/`googleAdsConversionLabel`/`metaPixelId`) at store-creation time, same as it does for `config/seo.ts` etc. — otherwise every newly created store silently gets no ad tracking until someone edits the file by hand.
- Webhook *registration* (creating the `orders/paid` webhook subscription in each store's Shopify custom app, pointed at `https://{store-domain}/api/webhooks/shopify/orders`, with the secret copied into that store's `SHOPIFY_WEBHOOK_SECRET`) is Shopify Admin configuration per store, not something this template can automate from inside itself.

typecheck/lint/build all green; build output confirms both `/api/webhooks/shopify/orders` and `/thank-you` register as dynamic routes.

## Verification & tooling notes (READ — saves you an hour)

- `npm run typecheck && npm run lint` green. **Do NOT run `npm run build` while the dev server is running** — they share `.next` and it 500s the dev server (restart it if you did).
- Dev server convention: port 3199 (`npm run dev -- --port 3199`; kill with `lsof -ti:3199 | xargs kill`).
- Screenshots: plain `chrome --headless --screenshot --window-size=390,…` silently clamps width to ~500 → false layouts. Use the puppeteer-core scripts in the session scratchpad pattern: `shot.mjs <url> <out.png> <width> [height]` (puppeteer-core + Brave at `/Applications/Brave Browser.app/...`). An add-to-cart/cart-flow test script pattern also exists (`cart-flow-test.mjs`).
- `IconArrow` points RIGHT natively (ARCHITECTURE-DECISIONS was wrong, now corrected; carousel + pagination fixed).
- Streamed pages: `notFound()` after loading.tsx streams returns HTTP 200 + noindex meta + 404 UI — real 404 status comes with the custom not-found page (Phase D remainder / E).
- Recurring layout pitfall found 3×: intrinsically wide children (thumb rails, long text-logo) force grid/flex tracks past the viewport — fix is `min-w-0` on the grid/flex item (see product-view gallery, cart-page grid, header logo truncate).
