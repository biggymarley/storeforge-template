# StoreForge Template Contract

Read this before touching design, layout, or component code in this repo ‚Äî or in
any sibling template repo StoreForge (the local bulk-store tool at
`~/Desktop/store-bulk`) will clone. Breaking it doesn't fail loudly here; it fails
silently later, on someone else's live store, during an automated update.

## Why this matters

StoreForge generates real stores from this repo by cloning a tagged version and
injecting `config/*.ts` + `public/branding/*` per customer. When you tag a new
version here, StoreForge's update engine deletes every template-owned file in each
live store and replaces it with the new tag's files ‚Äî except for a fixed list of
"protected paths" it never touches. If this repo drifts from the contract below,
either (a) new stores break at creation time, or (b) old stores break at their next
update, and nobody notices until a customer's site is down.

## The non-negotiable contract

Full version lives in `TEMPLATE-BUILD-SPEC.md` ¬ß2 ‚Äî this is the summary you
actually need day to day.

1. **Store-specific values live only in:** `config/store.ts`, `config/legal.ts`,
   `config/seo.ts`, `config/content.ts`, `public/branding/`, `store.meta.json`,
   `.env`/`.env.*`. Nothing else may contain a literal brand name, hex color,
   social URL, or company detail. A component that needs a store value goes
   through `resolveStoreConfig()` / `resolveLegalConfig()` / `resolveSeoConfig()`
   / `resolveContentConfig()` (`lib/config.ts`) ‚Äî never a raw import of
   `config/store.ts`, never a hardcoded fallback duplicated somewhere else.

2. **`template.json`'s `protectedPaths` is the source of truth for what
   StoreForge will never overwrite on an update:**
   `["config/", "public/branding/", "store.meta.json", ".env", ".env.*"]`
   Don't move a template-owned file into one of these paths, and don't take a
   protected path out of it, without updating this list *and* flagging it ‚Äî
   store-bulk's `render-config.ts` and this repo both have to agree on it.

3. **Config field changes must be backward-compatible.** A live store's
   `config/store.ts` was string-generated by StoreForge at whatever template
   version existed when the store was created or last had its branding edited.
   It's a protected path, so a template update never regenerates it. So:
   - New fields: add them optional in `lib/types/config.ts`, give them a default
     in the matching `resolve*Config()` function. Never assume a field exists
     directly on the imported config object.
   - Never rename or remove a field that a `resolve*Config()` currently reads ‚Äî
     that breaks every store created before today.
   - A genuinely breaking config shape change is a major version bump you flag
     to the human, not something to do silently inside a patch tag.

4. **Versioning:** bump `template.json.version`, `git tag vX.Y.Z`, push tags.
   StoreForge's `/template` page and update engine key off git tags, not branch
   state ‚Äî nothing is "released" until it's tagged.

5. **Before tagging:** `npm run typecheck`, `npm run lint`, `npm run build` all
   clean. Flip `config/store.ts` to a deliberately different palette/name and
   eyeball every page ‚Äî fastest way to catch a value that snuck out of config/.

## Current state of this repo (as of 2026-07-19) ‚Äî don't regress these

- Header (`components/layout/header.tsx`) renders the store's actual logo image
  via `resolveStoreConfig().logo` (`next/image`), not just plain text. `logo.src`
  / `logo.alt` already flow end-to-end from store-bulk's `render-config.ts`
  through `config/store.ts`. Any nav/header redesign must keep reading through
  `resolveStoreConfig()` ‚Äî don't reintroduce a text-only or hardcoded logo.
- There is no search feature. `/search`, `/api/predictive-search`,
  `components/layout/search-bar.tsx`, the search GraphQL queries
  (formerly `lib/shopify/queries/search.ts`), `searchProducts`/`predictiveSearch`
  (`lib/shopify/api.ts`), and the `/search` sitemap entry were deliberately
  removed. Don't re-add a header search icon/bar or restore those files as a
  side effect of a nav or PLP redesign ‚Äî if search comes back, it's a deliberate
  feature decision, not a redesign default.
- PLP sort/filter infra (`lib/plp.ts`, `filter-panel.tsx`, `sort-select.tsx`) is
  shared by `/products` and `/collections/[handle]` and is unrelated to the
  removed search feature ‚Äî safe to keep using as-is.

## Landing page (`/`) redesign specifically

`app/(store)/page.tsx` reads `resolveStoreConfig().hero` (image/headline/subtext/
stats) and `resolveContentConfig()` (testimonials/brands/productReviews). A new
look for `/` should keep consuming these same resolved shapes ‚Äî restyle freely,
but if the new design needs a hero field that doesn't exist yet (a video
background, a secondary CTA‚Ä¶), add it the backward-compatible way (rule 3)
rather than hardcoding it.

## If you're building a brand-new template repo (not redesigning this one)

Same contract applies from day one ‚Äî copy this file and `TEMPLATE-BUILD-SPEC.md`
into that repo's root and build against them.

One thing to flag to the human before you start: store-bulk currently points at
exactly one `TEMPLATE_REPO` in its `.env.local`. A second, differently-designed
template repo needs either a manual env swap before creating stores with it, or a
change to store-bulk itself to support multiple templates side by side. Don't
assume multi-template support already exists ‚Äî it isn't built.
